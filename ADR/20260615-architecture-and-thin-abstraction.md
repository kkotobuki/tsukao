# Life GPS のアーキテクチャと薄い抽象方針

**ステータス**: Accepted (2026-06-15) / 決定1は `20260617-mobile-frontend-and-single-platform.md` により一部 Superseded（フロントは Next.js から Expo へ、AI 中継は Route Handlers から Supabase Edge Functions へ変更）。決定2（Supabase）・決定3（Anthropic 直叩き・薄い抽象）は有効。

# Motivation

## 問題

Life GPS は「自分専用の将来シミュレータ」であり、開発者本人が唯一のユーザー兼保守者である。実装を始めるにあたり、以下を確定しないと先に進めない。

1. **フロント/バックエンドの土台**: UI（双六的な未来の可視化・what-ifサンドボックス）と、AI呼び出しを含むサーバ処理の両方を1つの体系で扱いたい
2. **AI統合の抽象レベル**: シミュレーション結果をもとにした相談（AI相談層）で LLM を呼ぶ。ここに LangChain / LangGraph のようなオーケストレーション系ミドルウェアを挟むか、API を直接叩くかで、保守性とデバッグ容易性が大きく変わる
3. **データストアと認証**: 入力値（年齢・年収・資産・貯蓄額）、提示後の主観反応ログ、自己モデル（欲求・制約）を保存する先

個人運用・1人保守が前提のため、**多人数開発を想定した重厚な抽象層は、デバッグ困難・バージョン追従コスト・ベンダーロックインという形で負債になりやすい**。

## 目的

1. フロントとバックを単一フレームワークで完結させ、1人で保守できる構成にする
2. AI 呼び出しは、モデル差し替え・プロンプト調整・挙動デバッグが**直接効く**薄い構成にする
3. 認証・DB・将来の永続化（主観反応ログ等）を低運用負荷の単一基盤に寄せる

# Scope / Out of Scope

## Scope

- フロントエンド/バックエンドのフレームワーク選定
- AI（Anthropic API）統合の抽象レベルの方針
- データストア・認証基盤の選定
- オーケストレーション系ミドルウェアを採用しない方針の明文化

## Out of Scope

- 具体的なテーブルスキーマ（入力・主観反応ログ・自己モデルの物理設計）
- デプロイ先・ホスティングの詳細（別途判断）
- 認証フローの詳細（個人運用前提のため当面は最小構成）
- UIコンポーネント設計・状態管理ライブラリの選定
- シミュレーションモデルそのものの設計（→ `20260615-simulation-model.md`）

# 結果

## Pros

- Next.js に寄せることで、UI と AI を含むサーバ処理（Route Handlers）が同一プロジェクトで完結する
- Supabase により認証・DB・永続化を1つの基盤に集約でき、個人運用の管理対象が減る
- Anthropic API 直叩きにより、プロンプト・モデル・リトライ挙動がコード上で完全に見え、デバッグが容易
- オーケストレーション系を避けることで、ライブラリのメジャー更新追従や内部挙動のブラックボックス化を回避できる
- 自己モデル（欲求・制約）をシミュレーションと相談層に接続する独自ロジックを、フレームワークの作法に縛られず素直に書ける

## Cons

- マルチステップのエージェント的処理（ツール呼び出しの連鎖等）を将来導入する場合、薄い抽象だと配線を自前で書く必要がある
- Supabase へのベンダーロックインが発生する（ただし Postgres ベースのため移行余地は残る）
- 直叩きゆえ、プロンプトのバージョン管理・観測・評価の仕組みは自前で用意する必要がある

# 決定 & 結論

## 1. フロント/バックエンドは Next.js を採用する

**決定**: UI とサーバ処理（AI 呼び出し・シミュレーション計算）を Next.js に集約する。AI を呼ぶサーバ側処理は Route Handlers で実装する。

個人開発で UI とサーバを別スタックに分けると保守対象が倍になるため、単一フレームワークで完結させることを優先する。双六的な未来可視化・what-ifサンドボックスといったインタラクティブUIと、サーバ側のシミュレーション/AI処理が同居できる点を重視した。

## 2. データストア・認証は Supabase を採用する

> ⚠️ **v1 では見直し**: `20260617-mobile-frontend-and-single-platform.md` の追補(2026-06-19)により、v1 のデータストアは **Notion** とし Supabase は採用見送り（個人アプリの範疇を超えたら移行）。下記は移行先としての評価として有効。

**決定**: 入力値・主観反応ログ・自己モデルの永続化、および（必要になれば）認証を Supabase に集約する。

唯一のユーザーが本人であっても、入力履歴・主観反応の時系列ログは「自己参照的充足」を測る中核データ（→ `20260615-self-referential-satisfaction-core.md`）であり、永続化先が要る。Postgres ベースで将来のクエリ・分析がしやすく、認証込みで運用負荷が低い点を評価した。

## 3. AI は Anthropic API を直叩きし、オーケストレーション系ミドルウェアは使わない（薄い抽象）

**決定**: AI相談層（→ `20260615-ai-advisory-layer.md`）の LLM 呼び出しは Anthropic API を直接呼ぶ。LangChain / LangGraph 等のオーケストレーションフレームワークは採用しない。抽象は「自前の薄いラッパ1枚」までに留める。

### 比較検討した選択肢

| 候補 | 採用しなかった理由 |
|---|---|
| LangChain / LangGraph | マルチエージェント・ツール連鎖の抽象は強力だが、個人開発には過剰。内部挙動がブラックボックス化しデバッグが難しく、メジャー更新の追従コストも高い |
| Vercel AI SDK | Next.js との相性は良くストリーミング等が楽だが、本用途のAI相談は「自己モデルに接続して選択肢をシミュレーションに投入する」独自ロジックが中核で、SDKの抽象に乗せる利得が薄い。将来必要になれば薄く部分採用する余地は残す |
| Anthropic API 直叩き（採用） | プロンプト・モデル・リトライがコード上で完全に可視。自己モデル接続という非汎用ロジックを素直に書ける。ロックインも最小 |

### 「薄い抽象」の境界線

- 許容する抽象: API キー管理・リトライ・型付きレスポンス整形程度の**自前ラッパ1枚**
- 避ける抽象: 思考の連鎖を隠蔽するエージェントフレームワーク、プロンプトをDSL化する層、暗黙のメモリ管理
- もし将来マルチステップ処理が本当に必要になったら、その時点で改めてADRを起こして再評価する（先回りして重い抽象を入れない）
