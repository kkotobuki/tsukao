# 使い切り視点（寿命ゴール・使い切りペース線・性別任意入力）

**ステータス**: Accepted（2026-07-03）

**読者**: ツカオ（Life GPS）の実装者（本人）。**目的**: 資産推移グラフの問いを「足りるか？」から「寿命までに使い切れるか？」へ反転させた判断と、その前提（予測寿命の算出・性別入力・2モード切り替え）の根拠を記録する。`20260615-simulation-model.md` の「N年レバー（既定30年）」と「チャート100歳打ち切り」を supersede する。

# Motivation

## 問題

資産推移グラフは 100歳固定で線を引いており、余裕がある人では**線が下がらずに増え続けるだけ**だった。本人の実感として：

- 「お金が足りすぎているとは分かるが、永遠に続く線になっていて分かりにくい」
- 終端（100歳）が恣意的な点で、「結局どれくらい余るのか」「だから今どうしていいのか」がグラフから読み取れない
- 「余る」という事実が「使っていい」という許可（プロダクトの背骨）に翻訳されていない

判定も `annualFreeSpendingYen >= 0` の二値のみで、「余裕が少しある」と「大幅に余る」が同じ緑表示に潰れていた。

## 目的

1. グラフに**ゴール（予測寿命）**を与え、線が終わる意味を作る
2. 視点を「資産がいくら残るか」から「**寿命までに使い切れるか**」へ反転する
3. 使い切りペース額を「今の生活費の◯倍」等の体感変換で返し、"使っていい" 許可を可視化する
4. 「死ぬ年齢は入力しない」方針は維持する（入力させると不安を煽る＋答えられない）

# Decision

## D1. グラフの終端＝予測寿命（統計から算出、入力しない）

100歳固定をやめ、終端を予測寿命にする。算出式（`src/core/data/life-expectancy.ts`）：

```
予測寿命 = round( 現在年齢 + 平均余命(現在年齢・性別) + 伸び率 × (退職を迎える年 − 2024) )
```

- **平均余命**: 厚労省 令和6(2024)年簡易生命表 表1「主な年齢の平均余命」。5歳刻みを線形補間
- **伸び率**: 社人研「日本の将来推計人口（令和5年推計）」死亡中位仮定。平均寿命 2020年 男81.56/女87.71 → 2070年 男85.89/女91.94 の線形換算（年あたり 男+0.087 / 女+0.085）
- 出生時の平均寿命でなく**現在年齢の平均余命**を基礎にする（現在年齢まで生き延びた条件付き期待値の方が正確）
- 「退職を迎える年」まで伸びを織り込む＝「その人が退職する頃の平均寿命」の近似（コホート寿命の簡易近似）
- 安全弁: 下限 退職年齢+1、上限105歳

## D2. 性別は任意入力（未回答なら男女平均）

平均余命の男女差は約6年あり、使い切り額の分母が1〜2割変わるため性別入力を追加する。ただし**任意**とし、未回答は男女平均で計算する。必須4項目（年齢・年収・資産・支出）は増やさない（`20260615-input-model-and-data-sources.md` の入力最小方針を維持）。

## D3. 退職後の線＝使い切りペース（寿命でちょうどゼロ着地）

余裕がある場合、退職後の消費を「**予測寿命でちょうど資産がゼロになる年間支出額**」にして線を引く。額は年金＋パート・退職後イベント・投資ポットの運用益を含む退職後シミュレーションを二分探索で解く（`solvePresentMonthlyHeadroomYen` と同型。閉じた式でなく再評価型なので運用益を落とさない）。

線は退職から下りながらゴール（予測寿命）でゼロに着く。**線そのものが「ここまで使っていい」の視覚表現**になる（記述的な平均一本線から、退職後のみ許可的な線へ半歩踏み出す）。

## D4. 2モード切り替え（使い切り／不足）

境界は「**使い切りペース額 ≥ 退職後の最低生活費**」：

| モード | 条件 | 見せ方 |
|---|---|---|
| spendDown | 使い切り額 ≥ 最低生活費 | 使い切り線＋「月◯円使えます。使い切れますか？」＋今の生活費の◯倍 |
| shortage | 使い切り額 < 最低生活費 | 最低生活費で取り崩し、尽きる年で打ち切る線＋改善提案（変更前と同じ） |

`hasRoom`（余裕あり/なし）の判定もこの境界に統一する。

## D5. N年レバーの廃止（残り年数＝寿命−退職年齢）

「貯蓄を使う年数 N（既定30・自由入力）」のスライダーを UI から外す。閉じた式（`annualFreeSpendingYen`、「今の許可」ソルバの判定に使用）の N は未指定なら「退職〜予測寿命」の年数を使う。`yearsToSpendSavings` は入力型に残す（テスト・後方互換の上書き用）。

# Rationale

- **なぜ寿命をゴールにするか**: 終わりのない線には「使い切る」という問いが立てられない。ゴールがあって初めて「余る＝使い残す」が可視化され、「使っていい」への翻訳が成立する。
- **なぜ入力させず統計で予測するか**: 「死ぬ年齢は入力しない」は SPEC 制定時からの方針（不安を煽る・本人に答えがない）。統計の平均なら平均一本線の思想（平均どおり進めばの基準線）と整合する。
- **なぜ伸びを織り込むか**: 静的な平均寿命は「今の高齢者」の数字で、若い利用者が高齢になる頃には寿命が延びている。過小な寿命は使い切り額の過大提示＝使いすぎの許可につながるため、保守側でなく正確側に寄せる。
- **なぜ性別を任意にするか**: 精度（男女差6年）と入力の軽さのトレードオフ。未回答＝男女平均のフォールバックがあれば必須にする必要がない。
- **なぜ2モードに分けるか**: 使い切りペースが最低生活費未満の人に「あなたは年◯円使えます」と見せると、生活が成立しない額を「使っていい」と誤読させる。不足側は従来の「何歳まで持つか＋打ち手」の方が誠実。
- **なぜ二分探索か（閉じた式でないか）**: 閉じた式（貯蓄÷N）は運用益・退職後イベントを落とす。既にある `simulate` 再評価＋二分探索のパターンを流用すれば、モデルを変えずに正確な「ゼロ着地額」が解ける。

# Consequences / Out of Scope・TODO

- 生命表・社人研推計はコード内の静的テーブル（`life-expectancy.ts`）。簡易生命表は毎年更新されるため、snapshot 再生成の仕組みができたら追従する（TODO）
- 使い切り線は「終端資産＝0」だけを拘束する。退職後イベントの一時費用が大きいと**途中で一時的にマイナス**になる経路がありうる（終端は0に戻る）。v1 では許容し、必要なら「全年 ≥ 0」制約に強化する
- shortage モードのグラフも終端は予測寿命まで（枯渇が先ならそこで打ち切り）。100歳という数字はコードから消えた
- 「今の許可」（`solvePresentMonthlyHeadroomYen`）は閉じた式ベースのまま。使い切りベース（現役の上乗せ後も寿命ゼロ着地を保つ額）への置き換えは別判断
