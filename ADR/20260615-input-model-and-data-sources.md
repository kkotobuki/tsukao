# 入力モデルと統計データの出所

**ステータス**: Accepted（入力設計・出所・ベース/差分の建て付けは確定 / 具体数値は Proposed）(2026-06-15)

**読者**: Life GPS の実装者（本人）。**目的**: シミュレーション（→ `20260615-simulation-model.md`）に供給する入力の設計（必須入力／任意トグル）と、各値の統計出所、ベース＝単身＋差分という建て付け、そして「平均値に何が内包されているか」を言語化して記録する。

# Motivation

## 問題

平均一本線モデル（→ `20260615-simulation-model.md`）を動かすには、「何を本人に入力させ」「足りない分をどの統計から引き」「トグルONのとき何をどう足すか」を確定する必要がある。ここには3つの課題がある。

1. **入力負荷**: 結婚・子供・住宅を最初から全部入力させると、これらへの欲求が薄い本人には摩擦が大きく離脱を招く
2. **二重計上**: 世間の「平均消費」は既婚者も独身者も混ざった平均であり、これをベースに結婚トグルで上乗せすると既婚分を二重計上する
3. **出所と限界**: 昇給・消費・物価・運用利回りはそれぞれ別の統計から引く。出所を明示しないと数値の根拠が追えず、またクロスセクション統計の限界（合成コホート）を見落とす

## 目的

1. 必須入力を最小（年齢・年収・現在資産・毎月の固定費）に絞り、人生の状態は任意○トグル＋progressive disclosure とする
2. 二重計上を避けるため、ベース消費は世帯パターン（単身/夫婦のみ）の直接選択とし、子供などその他のライフイベント費用は専用DBで個別に加算する
3. 全ての値の出所を表で明示し、平均に内包されているもの／いないものを言語化する

# Scope / Out of Scope

## Scope

- 必須入力と任意トグルの切り分け、progressive disclosure
- ベース消費＝世帯パターン（単身/夫婦のみ）、その他費用＝DBのライフイベント加算という建て付け
- 各値（昇給・消費・物価・税/社保・運用利回り）の統計出所の確定
- 平均値に内包される事象／内包されない事象の言語化
- クロスセクション統計の限界（合成コホート）の明記

## Out of Scope

- 具体的な数値（実質運用利回り・税率・各トグルの差分額）。実装時にソースから算出する
- 入力フォーム・トグルUIの画面設計
- 統計データの取り込み方式（手動投入/API/静的）

# 結果

## Pros

- 必須入力4項目で即「現在地」を提示でき、離脱を抑えられる
- ベース消費を世帯パターンで選び個別費用をDBで分離することで、各項目の影響を独立に理解でき、二重計上を避けられる
- 出所を表で明示することで、数値の根拠と再現性が確保される
- 内包項目を言語化することで、「何が見えていて何が見えていないか」が本人にも明確になる

## Cons

- 単身ベース＋差分は、現実の複合的な世帯（共働き・複数世代同居等）を単純化する
- クロスセクション統計の限界（合成コホート）は本質的に解消できず、注記で緩和するに留まる
- 統計データは年次・出典バージョンの管理が必要

# 決定 & 結論

## 1. 必須入力は4項目、人生の状態は任意○トグル（progressive disclosure）

**決定**: 必須入力は **年齢・年収（額面）・現在資産・毎月の固定費**（家賃・光熱費・ネット・携帯等）に絞る。結婚・子供・住宅・投資といった人生の状態は**任意の○トグル**とし、未設定なら「その状態でない」ものとして扱う。入力は progressive disclosure とし、まず現在地と平均的な見通しを出してから、関心に応じてトグルを開く。

「結婚・子供・住宅への欲求は薄め」という本人特性に配慮し、これらを入力必須にしないことが摩擦低減の要点である。

## 2. ベース消費は「世帯パターン」、その他のライフイベント費用は専用DBで加算

**決定**: 消費は2層で扱う。

1. **ベース消費＝世帯パターン**: 単身 / 夫婦のみ を家計調査から選ぶ。**結婚は「単身→夫婦のみ」への切替**として扱う（差分加算ではなくパターン選択）。
2. **その他のライフイベント費用**: 子供・住宅・自動車・離婚等は DB「ライフイベント別 年額コスト」で個別管理し、ON（任意選択）時にその**年額（＋初期費用）を加算**する。各イベントは固有の出所を持つ。

| 項目 | 扱い | 出所 |
|---|---|---|
| 結婚（世帯パターン） | 単身→夫婦のみへ消費ベースを切替 | 家計調査（単身 / 夫婦のみ） |
| 子供 | DBのライフイベント費用。1人あたり **120万円/年（養育費＋教育費込み）× 人数** を加算 | 国立成育医療研究センター（家計調査の世帯消費差分 43,033円/月は教育費が薄いため不採用） |
| 住宅・自動車等 | DBのライフイベント費用。年額（＋初期費用）を加算 | 各項目の出所（DBに記載） |
| 投資 | 資産に長期の**実質**運用利回りを乗せる（収入ではなく資産側に効く） | GPIF等 |

### Why（世帯パターン＋DB分離にする理由）

世間の平均消費は既婚・独身が混ざった値で、これをベースに結婚分を上乗せすると二重計上する。そこでベース消費は「単身 or 夫婦のみ」という世帯パターンの**直接選択**にして二重計上を避ける。一方、子供などの個別費用は世帯消費の差分では**教育費を十分に拾えない**ため、専用の出所（成育医療センター）を持つDBのライフイベント費用として分離管理する。これにより、家計調査由来の費用と個別出所の費用が混ざらず、各項目の根拠が追える。

## 3. 各値の統計出所を確定する

**決定**: 各値の参照源を以下に固定する。トグルや前提を提示する際は、対応する出所を本人にも示す。

| 用途 | 出所 |
|---|---|
| 昇給カーブ（年齢×年収の推移） | 賃金構造基本統計調査（賃金センサス、厚労省） |
| ベース消費（世帯パターン: 単身 / 夫婦のみ） | 家計調査（総務省統計局。単身世帯・世帯類型別の内訳あり） |
| 子供のライフイベント費用（1人120万円/年・教育費込み） | 国立成育医療研究センター（参考: 文科省 子供の学習費調査） |
| 資産分布（参考・現在地の位置づけ） | 家計の金融行動に関する世論調査 |
| 物価上昇率（実質換算の参考。直接の上乗せには使わない） | 消費者物価指数（CPI、総務省統計局） |
| 額面→手取り変換（税・社会保険料） | 所得税・住民税の税率、社会保険料率 |
| 投資トグルの運用利回り | 長期の平均運用利回り（出典は実装時に確定） |

## 4. 平均値に「内包されているもの／いないもの」を言語化する

**決定**: 平均値モデルの透明性のため、ベース平均に何が含まれ、何が含まれないかを明記する。

### ベース平均に内包されているもの

- **平均的な医療費**（家計調査の「保健医療」費目）→ 病気は平均としてここに含まれる
- その年齢・世帯類型での**全消費費目の平均水準**（食料・住居・光熱水道・家具家事・被服・保健医療・交通通信・教育・教養娯楽・その他＝交際費やこづかい等）
- **平均的な昇進・昇給**（賃金カーブに織り込み済み）と**賞与**（年間賞与を含む年収ベース）

### ベース平均に内包されていない（別途処理・または対象外）もの

- **税金・社会保険料**: 消費支出には含まれない（家計調査では「非消費支出」として別枠）。賃金センサスの年収も額面。→ 額面→手取り変換で別途差し引く（決定3）
- **物価上昇**: 本モデルは物価を考慮せず**現在価値（今日の円）**で計算する（→ `20260615-simulation-model.md` 決定2）。名目の統計値（運用利回り等）を使う際は実質（物価控除後）に換算する
- **住宅ローンの元本返済**: 消費支出に含まれない（持家世帯の住居費は安く出る）。→ 住宅トグルで扱う際の注意点
- **失業・大幅な収入減（転職リスク）**: 就業者断面の統計から欠落（生存バイアス）。→ モデルでは無視（→ `20260615-simulation-model.md` 決定3）
- **個人のテールリスク**: 「自分にだけ大きな不運」は平均では表現されない（平均一本線の構造的帰結）

## 5. クロスセクション統計の限界を明記する

**決定**: 昇給カーブ・消費・資産分布は**ある一時点で異なる年齢層を横断観測したデータ**である。これを「年齢を追うごとの推移」に使うと、実在の個人の経路ではなく**合成コホート（synthetic cohort）**になる。以下のバイアスを、コード/データ層のドキュメントと、ユーザー提示の両面で明記する。

- **世代効果（コホート効果）**: 現在の高年齢層の年収・資産は、現在の若年層が将来同年齢に達したときの値とは限らない
- **生存バイアス**: 高年齢層の分布は、退職・廃業・脱落した層が抜けた「残った人」の分布になりやすい

これは平均一本線が確率的予測ではないこと（→ `20260615-simulation-model.md`）と一貫した、誠実な限界提示である。
