## 1. 破壊的操作の境界線（auto mode でも止まるべきもの）

Claude Code 標準のガードに加え、以下はユーザー明示指示時のみ実行する。

### 1.1 git
- `git push --force` / `--force-with-lease`（特に main / develop / release ブランチ）
- `git reset --hard`
- `git branch -D`（マージ前のブランチ削除）
- `git checkout .` / `git restore .`（未コミット変更の破棄）
- `git rebase -i`、hook スキップ（`--no-verify`, `--no-gpg-sign`）

### 1.2 ファイルシステム
- `rm -rf`、ディレクトリ単位の削除
- 大量ファイルの一括削除
- 既存ファイルの完全上書き（読まずに `Write` する等）

### 1.3 Docker / コンテナ
**確認なしで実行してよい**:
- `docker builder prune -af`（ビルドキャッシュ削除）
- `docker image prune`（ダングリング image 削除、`-a` は付けない）
- `docker ps -a` 等の状態確認系コマンド

**実行前に必ずユーザー確認**:
- `docker stop` / `docker rm` / `docker rmi` / `docker volume rm`
- `docker system prune -a`
- DB コンテナの停止・リセット系コマンド全般

### 1.4 データベース
- 本番への migration 適用（`prisma migrate deploy`, `supabase db push`, `alembic upgrade head` 等）
- DB リセット系コマンド（`db reset`, `DROP DATABASE`, `TRUNCATE` 等）
- 既存テーブルへの破壊的変更を含む migration の作成

### 1.5 コスト発生する操作
- LLM API のバッチ・ループ呼び出し（数十回〜の繰り返し）
- クラウドリソースの作成（DB インスタンス、ストレージ、関数、コンテナ）
- 課金される外部 API の連続呼び出し

## 3. シークレット・依存管理

### シークレット
- `.env*` / `credentials.json` / `*.pem` / `*.key` 等は絶対にコミットしない
- API キー・トークンをコード内にハードコードしない
- ログ・エラーメッセージにシークレットを出力しない
- **環境変数の値を Claude との会話（チャット）に貼ることを促さない**
  - 会話履歴・キャッシュ・テレメトリ経由で漏洩するリスクがあるため
  - 共有が必要な場合は「変数名」「存在するか」「型」のみを扱う
  - 値そのものが必要な処理は、ローカルファイル / `.env` 経由で読み込ませる

### 依存パッケージ
- 新規パッケージの追加はユーザー確認（用途・代替案の提示込み）
- メジャーバージョンアップは破壊的変更の可能性を調査してから提案
- 既存依存の削除・ダウングレードもユーザー確認

- PR サイズは適切に分割する（レビュアーが追える単位に）
- WIP / fixup コミットは整理してから PR を作成する

## 5. 削除・クリーンアップ提案の作法

- **「最終ログ時刻」「最終更新時刻」だけで unused 判定しない**（内部通信だけで動くサービス、cron でしか叩かれないジョブ、稀にしか動かないバッチがある）
- **「停止」と「削除」は別物として扱う**（後者はデータ消失の可能性）
- データを伴うリソース（DB volume, ストレージバケット, ログ等）の削除前にバックアップ要否を確認
- 「使われていない」と判断する前に、grep / コードリファレンス / デプロイ履歴を最低 1 つは確認する

### プロジェクト概要
ツカオ（旧称 Life GPS）— 「将来が見えると、今を楽しめる」をコンセプトにした将来家計シミュレータ。**将来のお金の不安で過剰に節約してしまう人**（まず開発者本人、そして同じ悩みの Z 世代）に、現在地と将来の見通しを見せ、「**今、経験にお金を使っていい**」という余裕（安心）を返す。アプリが「使え」と命令するのではなく、本人が自分の未来を見て「使っても大丈夫」と自分に許可を出せるようにする。年齢・年収・現在資産・毎月支出から現在価値（物価無視）で平均一本線＋ライフイベントの未来を描き、退職時資産を「今の生活費なら◯年分」に体感変換。中核思想は社会比較（順位・percentile・スコア）を排した「自己参照的充足」。AI相談層は断定せず選択肢を what-if に変換して結果を一緒に見る「相棒」。設計判断は `ADR/`、実装仕様は `SPEC.md`。

### ドメイン知識
- **自己参照的充足**: 他者比較（順位・percentile）ではなく「自分はこの未来で満足か」を中核アウトプットにする本プロジェクトの根本思想。順位・同年代比較は意図的に出さない。
- **平均一本線（決定論）**: 確率分岐（モンテカルロ）は採らず、統計の平均値で1本の見通しを引く。確率提示が不安を増幅する逆効果を避けるため。「平均どおり進めば」の基準線であり予測ではない。
- **トグル差分**: 結婚・子供・住宅・投資など「選んで宣言する人生の状態」を平均ベースに差分加算/減算する建て付け。ベースはトグル全OFF（単身・子なし・持家なし・投資なし）。
- **偶発リスクの線引き**: 病気＝ベース消費に内包 / 転職＝モデル化しない（生存バイアスで出典が裏づけられない）/ 暴落＝投資トグルの平均利回りに内包。
- **体感変換**: 絶対額でなく「今の生活費なら◯年分」等、本人の生活実感に紐づく相対表現に翻訳して提示する。
- **what-if サンドボックス**: 入力をいじると未来が即引き直される操作を中核UIにする。AI相談の選択肢もここの入力に変換される。
- **自己モデル**: 本人の欲求（何があれば満足か）・制約（できない/したくない）・主観反応ログの蓄積。AI相談層が参照して非汎用化する。
- **金商法グレー回避**: 特定銘柄・「これで儲かる」式の断定的投資助言は行わない。提示は本人が挙げた選択肢のシミュレーション上の帰結に限定（設計上の回避方針であり正式な法務判断は別途）。
