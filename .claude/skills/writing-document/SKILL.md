# ドキュメント記述ガイドライン

ドキュメントを作成・編集する際に従うガイドライン。
長期的に価値を持ち続けるドキュメントを書くための原則と実践をまとめている。

---

## 前提: 情報が不足している場合

ドキュメントに記述すべき **Why / 背景 / 代替案 / 制約条件** のいずれかについて、
**複数の選択肢が考えられて一意に決まらない場合**は、必ず `/hearing` スキルを使用してユーザーからヒアリングを行う。
コードや既存ドキュメントから一意に決まる場合のみ、それを根拠に書き起こす。
複数候補が残ったまま推測で書かない。

---

## 核心原則: Whyを中心に書く

コード（What）は頻繁に変わるが、意思決定の理由（Why）は変わりにくい。
ドキュメントの長期的な価値は「なぜそうしたのか」にある。

### 記述すべき3層

| 層 | 内容 | 例 |
|----|------|----|
| **What** | 何をしたか・何があるか | 「認証にJWTを使用している」 |
| **Why** | なぜそうしたか | 「セッションストアの運用コストを避けるため、ステートレスなJWTを選択した」 |
| **Why not** | なぜ他を選ばなかったか | 「セッション方式も検討したが、水平スケール時のセッション共有が複雑になるため見送った」 |

Whatだけのドキュメントはコードを読めば分かる。**WhyとWhy notがドキュメント固有の価値になる。**

### 実践

- 方針・設計・技術選定には必ず理由を添える
- 検討した代替案がある場合、却下理由とセットで記録する
- 制約条件（技術的制約、ビジネス要件、時間的制約）を明記する

---

## 原則1: 時間に依存しない表現を使う

ドキュメントは書いた瞬間から過去になる。時間依存の表現は陳腐化の原因になる。

### 第一原則: そもそも時間参照を消す

ほとんどの場合、時間参照は不要。状態として記述すれば陳腐化しない。

| 避ける表現 | 書き直し |
|---|---|
| 「現在は Redis を使用」 | 「Redis を使用している」 |
| 「最近認証を導入」 | 「認証を導入している」 |
| 「今後 GraphQL 移行予定」 | 「GraphQL 移行は未着手」 |
| 「新しい〜」「従来の〜」 | 修飾語を削除して対象だけ書く |

### 時間参照が必要な場合: 不変の参照点を使う

「いつそうなったか」が意思決定の根拠として重要な場合のみ、以下のいずれかで書く:

| 参照点 | 例 | 適する場面 |
|---|---|---|
| バージョン番号 | 「v2.0で導入」「v1系では〜」 | SemVer 管理されたコード |
| 絶対日付 | 「2026-05-22の決定」 | バージョン管理外（メモ・運用手順） |
| イベント名 | 「Phase 2 で追加」「α版時点では〜」 | 名前付きマイルストーンがある場合 |
| 意思決定アンカー | 「ADR-007で決定」「PR #123 で導入」 | ADR/RFC/PR運用がある場合 |

### 避ける表現の一覧

| 避ける表現 | 理由 |
|---|---|
| 「現在は〜」「今は〜」 | 読む時期によって "現在" がずれる |
| 「新しい〜」「最近〜」 | "新しい" の基準が陳腐化する |
| 「今後〜予定」 | 実装済みなのか未着手なのか不明になる |
| 「従来の〜」「古い〜」 | 比較相手が変わると意味が崩れる |

---

## 原則2: 読者とゴールを冒頭で明確にする

ドキュメントの冒頭で「誰のための」「何を目的とした」文書かを宣言する。
読者が自分に必要かどうかを瞬時に判断でき、書き手も範囲を絞れる。

悪い例:
```markdown
# 認証機能について
認証機能の実装方法を説明します。
```

良い例:
```markdown
# 認証機能の設計
認証方式としてJWTを採用した背景と、検討した代替案を記録する。
```

---

## 原則3: 小さく、モジュラーに保つ

巨大なドキュメントは更新されなくなる。1つのドキュメントは1つのテーマに絞る。

- 長くなる場合はファイルを分割し、リンクで繋ぐ
- 1つの意思決定に複数の論点がある場合でも、セクションで明確に分離する

---

## 原則4: 事実と意見を区別する

事実（客観的に検証可能なこと）と意見（判断・評価）を混在させない。

悪い例:
```markdown
このライブラリは使いにくいのでAxiosに変更した。
```

良い例:
```markdown
fetch APIではリクエストのインターセプトやリトライを自前で実装する必要があり、
既存のAxiosラッパーと統一するためAxiosを採用した。
```

---

## 原則5: 暗黙の前提を明文化する

チーム内で「当然」とされている前提こそ、ドキュメントに書く価値がある。
新しい読者や将来の自分にとって、暗黙知は存在しないのと同じである。

記録すべき暗黙の前提の例:
- 「このテーブルのデータ量は最大でも数百件なのでインデックスは不要」
- 「この処理はバッチで夜間に実行されるためレイテンシは問題にならない」
- 「この仕様はiOSアプリとの互換性のために維持している」

---

## アンチパターン

以下のパターンに気づいたら修正する。

| パターン | 問題 | 対処 |
|----------|------|------|
| Whatだけ書いてWhyがない | コードと同じ情報しかなく、変更時に即陳腐化する | 理由・背景・代替案を追記する |
| 時間依存の表現 | 読む時期によって意味が変わる | 具体的なバージョンや状態で記述する |
| 読者不在 | 誰向けか不明で、情報の粒度が定まらない | 冒頭で読者とゴールを宣言する |
| 結論だけで過程がない | 意思決定を再検討できない | 検討過程と却下した選択肢を残す |
| 曖昧な表現 | 認識齟齬の原因になる | 具体的な数値・名称・条件で記述する |
