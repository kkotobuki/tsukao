現在のブランチの変更内容から PR を作成してください。
作成前に品質チェック（軽量自動チェック + マルチエージェントレビュー）を実行し、問題がなければ PR 作成に進みます。

---

## 事前準備

最初に `git fetch origin develop` を実行し、リモートの develop を最新化する。
以降の差分比較はすべて **`origin/develop`** を基準とすること（ローカルの develop は古い可能性がある）。

---

## Phase 1: 軽量自動チェック（必須・順次実行）

1. **ブランチ状態**: `git status` で未コミット変更・未追跡ファイルを確認
   - 会話履歴を遡って変更意図が明らかなものは対応する形でコミット
   - 意図が不明な変更はユーザーに確認
2. **コミット履歴**: `git log origin/develop..HEAD --oneline` で WIP / fixup コミットがないか確認
3. **Lint**: プロジェクトの lint コマンドを実行
   - 実行コマンドは CLAUDE.md / package.json の scripts から判別する
4. **Typecheck**: プロジェクトの typecheck コマンドを実行
   - TypeScript の場合は `tsc --noEmit` 系、その他言語は対応するもの
   - typecheck が定義されていないプロジェクトではスキップ可
5. **Build**: プロジェクトの build コマンドを実行

**いずれか失敗した場合**: PR 作成を中断し、修正を促す。修正後に再度本コマンドを実行する。

---

## Phase 2: マルチエージェント並列レビュー（必須）

`git diff origin/develop...HEAD` の差分を取得し、以下の観点ごとに **Agent ツールでサブエージェントを並列起動** する。

| エージェント | 観点 | 起動条件 | 方針 |
|------------|------|---------|------|
| A | コード品質・衛生 | 必須 | デバッグコード残存、不要ファイル混入、コンフリクトマーカー、シークレット漏洩、`console.log` 残存を検出 |
| B | ライブラリのベストプラクティス | 必須 | 差分内の import からライブラリを特定し、**新規・既存問わず WebSearch で最新の推奨パターン・deprecation・脆弱性を検証**。公式ドキュメント URL を添えて報告 |
| C | 影響範囲・整合性 | 必須 | 変更された関数・型の参照箇所を Grep で特定し後方互換性を確認 |
| D | セキュリティ | 条件付き | API エンドポイント・認証認可・データアクセス層の変更時のみ。認証ガード、OWASP Top 10、入力バリデーション、DB 認可ポリシーを確認 |
| E | パフォーマンス | 条件付き | DB クエリ・レンダリング最適化の変更時のみ。N+1、不要な再レンダリング、クエリの効率性を確認 |

### 結果レポート

各エージェントの指摘を **重要度の高い順** に統合して報告する。

| 重要度 | 基準 | 判定影響 |
|--------|------|---------|
| Critical | 本番障害・セキュリティインシデント直結 | PR 作成中断 |
| High | ビルド失敗・仕様違反・互換性破壊 | PR 作成中断 |
| Medium | 品質低下だが機能的には問題なし | PR 作成可（修正推奨） |
| Low | より良い実装の提案 | PR 作成可（任意） |
| Info | 参考情報 | 対応不要 |

各指摘には以下を付記する：

- **エージェント名** と **対象ファイル:行番号**
- **確信度**（高: ツール出力で客観確認 / 中: コード読解に基づく推測 / 低: 一般論ベース）
- **推奨対応**: 具体的な修正例、確認依頼、または情報提供のいずれか
- 確信度「中」「低」の場合は **誤検知の可能性がある旨** を明記

### 中断条件

- Critical / High（確信度「高」）が 1 件でもある → 修正してから再実行
- Critical / High が確信度「中」「低」のみ → ユーザーに判断を仰ぐ
- Medium 以下のみ → Phase 3 に進む

---

## Phase 3: PR 作成

1. push 済みでない場合は `git push -u origin <ブランチ名>` を実行
2. `git log origin/develop..HEAD --oneline` と `git diff origin/develop...HEAD` で最終差分を把握
3. PR タイトルは 70 文字以内、Conventional Commits 形式（`feat:`, `fix:`, `refactor:` 等）
4. `gh pr create --base develop` で PR を作成する。本文は HEREDOC で渡す

### PR 本文の生成ルール

以下の構成で PR 本文を生成する:

```markdown
## 概要

- 変更の目的と背景を 1-3 行で記述

## 変更内容

- 主な変更点を箇条書きで記述

## 動作確認

- [ ] {変更内容に応じた具体的な確認項目}

## チェック項目

- [x] このPRは `/create-pull-request` で作成した
- [x] 軽量自動チェック（lint / typecheck / build）パス
- [x] マルチエージェントレビュー実施済み
```

- 各セクションはコミットメッセージや差分から読み取れる情報で埋める
- 推測で補えない情報はユーザーに確認してから埋める
- 「動作確認」のチェックリストは変更内容に応じて具体的な項目を生成する

---

## PR 作成後

PR 作成後、以下のメッセージをユーザーに表示する：

> PR を作成しました。レビュアーへの依頼に進んでください。
