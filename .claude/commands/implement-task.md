# タスク実装支援

## 入力

引数: $ARGUMENTS

引数にタスクの説明を受け取る（例: `検索APIの実装`）。
引数が不明確な場合は、ユーザーに「何を実装しますか？」と確認する。

---

## 重要: ヒアリングの原則

**必ず1つずつ質問すること。** 複数の質問を一度にしない。

---

## ワークフロー

### ステップ0: ワークフローの todo リスト作成

コマンド開始時に以下の todo リストを作成し、進捗を追跡する:

1. タスクの特定と理解
2. ブランチのヒアリング
3. 既存コードの把握
4. ヒアリング（質問があれば）
5. 要件の整理と確認
6. コーディング
7. 完了チェックと結果報告

各ステップを開始する際に `in_progress` に、完了したら `completed` に更新すること。

### ステップ1: タスクの特定と理解

1. 引数を解釈してタスクを特定
2. 作業の目的と位置付けを理解する

タスクの概要をユーザーに説明する。

### ステップ2: ブランチのヒアリング

「最新の develop ブランチから切ります。よろしいですか？（別のブランチから切る場合はブランチ名を指定してください）」

回答を受けて:
1. ユーザーが承認 → develop から切る
2. ユーザーがブランチ名を指定 → そのブランチから切る
3. ブランチ名は `feature/{説明}` 形式とする（例: `feature/add-input-validation`）

```bash
git fetch origin {ベースブランチ}
git checkout -b feature/{説明} origin/{ベースブランチ}
```

### ステップ3: 既存コードの把握

実装に関係のある既存コードを、**Explore エージェント**（コード探索専用のサブエージェント）で把握する。

確認すべき観点:
- 関連するファイル・モジュールの構造
- 既存の実装パターン
- 依存関係
- テストの書き方

### ステップ4: ヒアリング

不明点を一問ずつ質問する。

質問の観点:
- 仕様の曖昧な部分
- エッジケースの扱い
- 既存コードとの整合性
- 設計上の判断が必要な箇所

質問がなければ「特に確認事項はありません」と伝えてステップ5へ進む。

### ステップ5: 要件の整理と確認

ヒアリング完了後、以下を整理してユーザーに確認を取る:

- 実装する内容の要約
- 明確になった仕様・判断事項
- 実装アプローチ

「この内容で実装を進めてよいですか？」

### ステップ6: コーディング

要件に従って実装を行う。

- タスクに対応するテストも同じタスク内で実装する
- PR サイズは適切に分割する（CLAUDE.md のブランチ / PR ルールに従う）

### ステップ7: 完了チェックと結果報告

以下の完了チェックを順に実行する:

1. **Lint**: プロジェクトの lint コマンドを実行
2. **Typecheck**: プロジェクトの typecheck コマンドを実行（定義されていなければスキップ可）
3. **Build**: プロジェクトの build コマンドを実行

実行コマンドは CLAUDE.md / `package.json` の scripts 等から判別する。

すべてのチェックが通ったら、結果を報告する:

```
## 実装完了報告

### 実装内容
- {実装した内容の要約}

### 変更ファイル
- {変更したファイル一覧}

### テスト
- {追加・修正したテスト}

### 確認事項
- [ ] lint パス
- [ ] typecheck パス
- [ ] build パス
```

---

## 注意事項

- PR 作成は `/create-pull-request` で行う（lint / typecheck / build / マルチエージェントレビューも統合されている）
- 実装中に方針から逸脱する必要がある場合は、ユーザーに確認を取る
